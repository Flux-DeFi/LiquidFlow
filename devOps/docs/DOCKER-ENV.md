# Environment Variables Reference

LiquidFlow uses three `.env` files located in `devOps/`:

| File | Purpose | Commit? |
|------|---------|--------|
| `.env.example` | Template with every variable documented | ✅ Yes |
| `.env.local` | Local Docker development — safe defaults | ✅ Yes |
| `.env.prod` | Production overrides — contains real secrets | ❌ No |

---

## PostgreSQL

Consumed by the `postgres` service in `docker-compose.yml`.

| Variable | Example | Description |
|----------|---------|-------------|
| `POSTGRES_USER` | `postgres` | Database superuser username |
| `POSTGRES_PASSWORD` | `postgres` | Database superuser password |
| `POSTGRES_DB` | `liquidflow_dev` | Name of the default database created on first start |

---

## Backend — Prisma & App

Consumed by the `backend` Node.js service.

| Variable | Example | Description |
|----------|---------|-------------|
| `DATABASE_URL` | `postgresql://user:pass@postgres:5432/db` | Full Prisma connection string. The hostname must be `postgres` (Docker service name) inside the container, `localhost` outside. |
| `NODE_ENV` | `development` | Runtime environment. Controls Prisma query logging and other behaviours. Values: `development` \| `production` |
| `PORT` | `3001` | TCP port the Express server listens on |
| `LOG_LEVEL` | `info` | Winston log level. Values: `error` \| `warn` \| `info` \| `debug` |

## AWS — Authentication

Used by the AWS CLI, Terraform provider and scripts. Pick **one** authentication method.

### Option A — Named profile (recommended for local dev)

```dotenv
AWS_PROFILE=liquidflow-dev
AWS_REGION=us-east-1
```

Requires `~/.aws/config` and `~/.aws/credentials` set up via `aws configure --profile liquidflow-dev`.

### Option B — Static credentials (CI/CD)

```dotenv
AWS_ACCESS_KEY_ID=AKIAIOSFODNN7EXAMPLE
AWS_SECRET_ACCESS_KEY=wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
AWS_REGION=us-east-1
```

> In production the EC2 instances authenticate via an **IAM Instance Profile** — no static keys needed on the server.

| Variable | Description |
|----------|-------------|
| `AWS_PROFILE` | Named profile in `~/.aws/config` |
| `AWS_ACCESS_KEY_ID` | Static IAM access key (CI/CD only) |
| `AWS_SECRET_ACCESS_KEY` | Static IAM secret key (CI/CD only) |
| `AWS_REGION` | AWS region for all CLI and Terraform operations |

## Terraform Scripts

Consumed by `devOps/terraform/scripts/bootstrap.sh` and `deploy.sh`.

| Variable | Example | Description |
|----------|---------|-------------|
| `PROJECT_NAME` | `liquidflow` | Used to name the remote state S3 bucket and DynamoDB table |
| `TF_LOG` | `INFO` | Terraform log verbosity. Values: `TRACE` \| `DEBUG` \| `INFO` \| `WARN` \| `ERROR` \| `OFF` |

### Sensitive Terraform Input Variables

Variables prefixed with `TF_VAR_` are automatically picked up by Terraform as variable overrides, keeping secrets out of `terraform.tfvars`.

| Variable | Terraform var | Description |
|----------|---------------|-------------|
| `TF_VAR_db_username` | `db_username` | RDS PostgreSQL master username. Marked `sensitive = true` in `variables.tf`. The password is auto-generated by Terraform using `random_password` and stored in AWS Secrets Manager — it never appears in any env file. |
