# Terraform Deployment Guide — LiquidFlow

## Architecture Overview

```
Internet
    │
    ▼
┌─────────────────────────────────────┐
│         Application Load Balancer   │  (Public Subnets — 2 AZs)
│         HTTP → HTTPS redirect       │
└────────────────┬────────────────────┘
                 │
    ┌────────────▼────────────┐
    │    Auto Scaling Group   │  (Private Subnets — EC2)
    │  Launch Template (LT)   │  IMDSv2 · gp3 encrypted EBS
    │  min:2 / max:6 (prod)   │  SSM · CloudWatch Agent
    └────────────┬────────────┘
                 │
    ┌────────────▼────────────┐
    │  RDS PostgreSQL 16      │  (Private Subnets — Multi-AZ in prod)
    │  Encrypted at rest      │  Secrets Manager · Performance Insights
    └─────────────────────────┘
```

## Directory Structure

```
devOps/terraform/
├── environments/
│   ├── dev/          ← 10.0.0.0/16 · t3.micro  · db.t3.micro
│   ├── staging/      ← 10.1.0.0/16 · t3.small  · db.t3.small
│   └── prod/         ← 10.2.0.0/16 · t3.medium · db.t3.medium  Multi-AZ
├── modules/
│   ├── vpc/          ← VPC, subnets (pub/priv), IGW, NAT GW, Flow Logs
│   ├── security/     ← SGs: ALB, EC2, RDS (isolated per resource)
│   ├── ec2/          ← Launch Template, ASG, ALB, CloudWatch scaling
│   ├── rds/          ← PostgreSQL, Secrets Manager, Enhanced Monitoring
│   └── s3/           ← Versioning, encryption, lifecycle, access logs
├── global/
│   └── iam/          ← EC2 role: SSM + CloudWatch + S3 + SecretsManager
└── scripts/
    └── bootstrap.sh  ← Creates S3 state bucket + DynamoDB lock table
```

## Step 1 — Bootstrap Remote State (Run Once)

```bash
cd devOps/terraform/scripts
chmod +x bootstrap.sh
AWS_PROFILE=your-profile ./bootstrap.sh
```

This creates:
- **S3 bucket**: `liquidflow-terraform-state-<ACCOUNT_ID>` (versioned, encrypted, no public access)
- **DynamoDB table**: `liquidflow-terraform-locks` (prevents concurrent state writes)

Then patch the `<ACCOUNT_ID>` placeholder in all `backend.tf` files:

```bash
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
find devOps/terraform/environments -name 'backend.tf' \
  -exec sed -i "s/<ACCOUNT_ID>/$ACCOUNT_ID/g" {} +
```

## Step 2 — Deploy an Environment

### Using deploy.sh (recommended)

```bash
# Make script executable
chmod +x devOps/scripts/deploy.sh

# Plan (always plan before apply)
./devOps/scripts/deploy.sh dev plan

# Apply
./devOps/scripts/deploy.sh dev apply

# Review outputs (ALB DNS, S3 bucket, etc.)
./devOps/scripts/deploy.sh dev output
```

### Manual steps

```bash
cd devOps/terraform/environments/dev

terraform init
terraform validate
terraform fmt -check
terraform plan -var-file=terraform.tfvars -out=dev.tfplan
terraform apply dev.tfplan
```

## Security Checklist

- [x] No hardcoded credentials — passwords generated by `random_password`, stored in Secrets Manager
- [x] IMDSv2 enforced on all EC2 instances (`http_tokens = required`)
- [x] RDS never publicly accessible (`publicly_accessible = false`)
- [x] RDS in private subnets only
- [x] All S3 buckets have Block Public Access enabled
- [x] Encryption at rest on RDS (storage_encrypted = true) and S3 (AES-256)
- [x] Encryption in transit enforced via S3 bucket policy (deny HTTP)
- [x] EC2 SG only accepts traffic from ALB SG (not from the internet)
- [x] RDS SG only accepts traffic from EC2 SG
- [x] IAM uses least privilege — scoped to specific ARNs
- [x] SSM Session Manager enabled — no SSH ports required

## Accessing DB Credentials

Credentials are stored in AWS Secrets Manager after RDS is created:

```bash
# Get the secret ARN from Terraform output
SECRET_ARN=$(terraform -chdir=devOps/terraform/environments/dev output -raw db_secret_arn)

# Fetch the credentials
aws secretsmanager get-secret-value \
  --secret-id "$SECRET_ARN" \
  --query SecretString \
  --output text | jq .
```

## EC2 Access via SSM (no SSH required)

```bash
# List running instances
aws ec2 describe-instances \
  --filters "Name=tag:Project,Values=liquidflow" \
            "Name=instance-state-name,Values=running" \
  --query "Reservations[].Instances[].{ID:InstanceId,AZ:Placement.AvailabilityZone}" \
  --output table

# Start a session
aws ssm start-session --target i-xxxxxxxxxxxx
```

## CI/CD Integration (GitHub Actions example)

```yaml
- name: Terraform Plan
  run: ./devOps/scripts/deploy.sh ${{ github.ref == 'refs/heads/main' && 'prod' || 'staging' }} plan

- name: Terraform Apply
  if: github.ref == 'refs/heads/main'
  run: ./devOps/scripts/deploy.sh prod apply --auto-approve
  env:
    AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
    AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
```

## Destroying an Environment

```bash
# Never destroy prod without approval
./devOps/scripts/deploy.sh dev destroy
```

> ⚠️ **Production**: RDS has `deletion_protection = true`. You must disable it in the console or via Terraform before destroy will succeed.
